
# Архитектура проекта мониторинга принтеров

Архитектура данного проекта представляет собой **асинхронную систему мониторинга состояния принтеров через протокол SNMP** с отправкой уведомлений в **Telegram**.  

Программа построена по **модульному принципу**, где каждый компонент отвечает за отдельный уровень логики системы.

*Для эмуляции принтеров в локальной сети рекомендуется использовать <a link= "https://www.ireasoning.com/download.shtml">SNMP Agent Simulator</a>*

---

# Основные компоненты архитектуры

## 1. Ядро и управление конфигурацией  
**GlobalInfo, ConfigReader**

- **ConfigReader** считывает настройки из файла `config.json`.
- **GlobalInfo** выступает в роли **центрального хранилища (Singleton)** для всех настроек:
  - токены Telegram
  - ID администраторов
  - интервалы проверки
  - список активных принтеров

Это позволяет **любому модулю получить доступ к актуальным параметрам системы**.

---

## 2. Объектная модель устройств  
**Device, Printer, BotUpdater**

- **Device** — базовая модель устройства.
- **Printer** — наследник `Device`, описывающий структуру данных принтера:
  - IP-адрес
  - параметры картриджей
  - текущий уровень тонера
  - максимальный уровень
  - порог предупреждения
  - SNMP OID для получения данных

**BotUpdater** выполняет две задачи:

- связывает конфигурацию с объектами
- инициализирует список принтеров
- обновляет глобальное состояние программы

---

## 3. Сетевой уровень и сбор данных  
**SNMPGetter**

- Использует библиотеку **pysnmp**
- Выполняет **асинхронные SNMP-запросы** к устройствам
- Получает данные по **OID**, определённым в классе `Printer`

---

## 4. Коммуникационный слой  
**TelegramListener, TelegramSendAlert**

### TelegramSendAlert
Отвечает за **отправку уведомлений** администраторам, например:

- низкий уровень тонера
- ошибки устройств

### TelegramListener
Отвечает за **обработку входящих команд** пользователей:

- выполнение команд бота
- отправка файлов логов
- управление системой

---

## 5. Логирование  
**Logger**

Используется `RotatingFileHandler`.

Особенности:

- автоматическая **ротация логов**
- предотвращение **неконтролируемого роста лог-файлов**
- сохранение истории событий системы

---

# Схема взаимодействия компонентов

Ниже представлена логическая схема движения данных внутри системы.

```

Запуск системы
    │
    ▼
**main**.py
    │
    ▼
MainLoop
    │
    ▼
BotUpdater.InitPrinters()
    │
    ▼
ConfigReader → чтение config.json
    │
    ▼
GlobalInfo → хранение конфигурации

```

---

## Цикл мониторинга

```

MainLoop
    │
    ▼
SNMPGetter
    │
    ▼
Запрос к принтерам по IP (SNMP)
    │
    ▼
Получение уровней картриджей
    │
    ▼
Сравнение с порогом warning из Printer

```

---

## Реакция системы

### 1. Низкий уровень тонера

```

Printer warning
    │
    ▼
TelegramSendAlert
    │
    ▼
Сообщение администратору

```

### 2. Команда пользователя

```

Telegram
│
▼
TelegramListener
│
▼
Обработка команды

```

---

## Обновление конфигурации без перезапуска

```

BotUpdater.UpdateConfig()
    │
    ▼
Перечитывание config.json
    │
    ▼
Переинициализация списка принтеров
    │
    ▼
Обновление глобального состояния

```

# Технологические особенности

## Асинхронность

Проект построен на:

- `asyncio`
- `aiohttp`

Это позволяет:

- опрашивать **множество принтеров одновременно**
- **не блокировать** работу Telegram-бота
- эффективно использовать сетевые ресурсы

---

## Динамическая конфигурация

Система поддерживает **обновление параметров во время работы**:

- переинициализация списка принтеров
- обновление конфигурации
- перезапуск логгера

Это выполняется через:

```

BotUpdater.UpdateConfig()

```

и **не требует остановки бота**.
